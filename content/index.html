    • Docker

    Idea:

    Take any Linux program, and put it in a container
    Such as webapps, data stores, command-line tools, services
    Deliver container to DEV, SAT, PRD and it run anywhere
    If Docker fails : scrap it and start a new docker (of course examine the failed instance later on to find out the root cause)
    Format and Environment

  • VM
    executes machine code
    environment = something that looks like a computer

  • JVM executes JVM bytecode
    environment = Java APIs

  • Container executes machine code
    environment = Linux kernel system calls interface

    • Container Vs Virtual Machine:

    Containers pros:

  • Containers have low overhead
  • Normal processes are running on top of the normal kernel
  • Context switch between containers = context switch between processes
  • No device emulation ( no extra code path involved)
  • Benchmark show no difference at all between containers and bare metal
  • Containers have a higher density
    • Container Vs Virtual Machine:

    Container cons:

  • Run on top of a single kernel – could be exposed to Kernel vulnerability can lead to full scale compromise
  • Normal processes are running on top of the normal kernel
  • Containers can share files, sockets, FIFOs, memory areas…
    • Container Vs Virtual Machine:

    VM pros:

  • Strong Isolation
  • Inter-VM communication must happen over the network
  • VM can run as non-priveledged processes on the host
    • Container Vs Virtual Machine:

    Analogy:

    Room dividers VS Brick Walls
    Functional Differences between Containers and VMs:

    VM Need a full OS and associated tools ( Backups, logging, periodic job, execution, remote access)
    Containers can go both ways:
    Machine container (runs init, cron, ssh, syslog,…. And the app)
    Application container (runs app and nothing else; relies on external mechanisms)
    VM has long lifecycle and slow to scale
    Containers are fast and cheap ( resource wise)

    Development Process Differences between Containers and VMs:

    VM:
    Not realistic to have 1 VM per component in DEV
    PRD has additional/different components
    Therefore they end up with different environments
    Development Process (containers)
    Runs tons of containers in DEV machines
    Build THE SAME container for DEV and PRD
